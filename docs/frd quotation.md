## 1. Introduction

### 1.1 Purpose

The **Quoting Engine** is a specialized, stateless microservice responsible for the real-time calculation of logistics shipping rates. It acts as the mathematical brain of the marketplace, aggregating raw costs from external carriers (simulated), normalizing data, applying business profit margins, and returning a finalized sell price to the Core Monolith.

### 1.2 Scope

**In Scope:**

* Receiving Quote Requests via gRPC.
* Validating Locations (UN/LOCODE).
* Fetching raw rates from the Carrier Simulator (REST).
* Currency Conversion & Normalization.
* Application of Volumetric Weight logic.
* Application of Dynamic Profit Margins.
* Caching of results in Redis.

**Out of Scope:**

* User Authentication (handled by Java Monolith).
* Order Booking/Reservation (handled by Java Monolith).
* Payment Processing.

## 2. System Interfaces

### 2.1 Input Interface (Internal API)

The service MUST expose a **gRPC Server** listening on port `50051`.

* **Service Definition:** `QuotingService`
* **Method:** `GetQuotes(QuoteRequest)`
* **Payload (Protobuf):**
* `transaction_id` (UUID): For tracing.
* `origin_locode` (String): e.g., "CNSHA".
* `destination_locode` (String): e.g., "ESBCN".
* `cargo_details` (Object):
* `equipment_type` (Enum): 20FT, 40FT, 40HC, LCL.
* `total_weight_kg` (Float).
* `total_volume_cbm` (Float).
* `commodity_type` (Enum): GENERAL, HAZMAT, PERISHABLE.


* `incoterms` (Enum): FOB, CIF, EXW.
* `currency_requested` (String): EUR, USD.



### 2.2 Output Interface (Internal API)

The service MUST return a `QuoteResponse` Protobuf message.

* **Payload:**
* `quotes` (List):
* `quote_id` (UUID): Unique ID generated by the Engine.
* `carrier_name` (String).
* `total_price` (Decimal).
* `currency` (String).
* `transit_time_days` (Int).
* `valid_until` (Timestamp).
* `breakdown` (Object): Base Freight, Surcharges, Fees.





### 2.3 External Interface (Carrier Integration)

The service MUST act as a **REST Client** to query the Carrier Simulator.

* **Protocol:** HTTP/1.1 (Keep-Alive).
* **Timeout:** Strict **2000ms** hard timeout per request.
* **Circuit Breaker:** Open state after 5 consecutive failures.

## 3. Detailed Functional Requirements

### 3.1 Input Validation & Master Data

**FR-QUOTE-001: Location Validation**
The system MUST validate the received `origin_locode` and `destination_locode` against the local Redis Cache (`REF_DATA:LOCATIONS`).

* **Logic:** If a code does not exist in Redis, return `INVALID_ARGUMENT` gRPC error. The service MUST NOT query the Core Database.

**FR-QUOTE-002: Cargo Constraints**
The system MUST validate physical constraints of the cargo.

* **Logic:**
* If `equipment_type == '20FT'` AND `weight > 28,000kg` -> Return Error "Max Payload Exceeded".
* If `equipment_type == '40FT'` AND `weight > 30,000kg` -> Return Error "Max Payload Exceeded".



### 3.2 Aggregation Logic (The Fan-Out)

**FR-QUOTE-003: Concurrent Carrier Querying**
The system MUST spawn a separate Go Routine (Green Thread) for each configured Carrier/Mock endpoint.

* **Performance:** Queries must execute in parallel. Sequential execution is strictly forbidden.
* **Synchronization:** Use `sync.WaitGroup` to wait for all routines to finish or timeout.

**FR-QUOTE-004: Carrier Selection Strategy**
The system MUST determine which carriers to call based on the route.

* **Logic:**
* If origin is in Asia (CN, JP, KR) -> Query Carriers A, B, C.
* If origin is in Europe (DE, ES, FR) -> Query Carriers D, E.
* *(For MVP: Query "Carrier Simulator" for all routes)*.



**FR-QUOTE-005: Resiliency & Timeouts**
The system MUST enforce a "Global Timeout Budget" derived from the context.

* **Logic:** If the context passed via gRPC has a 2s deadline, the internal HTTP client calls MUST have a deadline of `context.Deadline() - 50ms` (buffer for processing).

### 3.3 The Calculation Engine (Core Logic)

**FR-QUOTE-006: Currency Normalization**
The system MUST convert carrier rates to the `currency_requested` by the user.

* **Input:** Carrier returns 2000 USD. User wants EUR.
* **Process:** Fetch `RATES:EXCHANGE:USD:EUR` from Redis (e.g., 0.92).
* **Calc:** `2000 * 0.92 = 1840 EUR`.

**FR-QUOTE-007: Base Cost Calculation**
The system MUST aggregate all cost components from the Carrier response.

* **Formula:** `TotalCost = BaseFreight + BAF (Bunker) + CAF (Currency) + ISPS (Security)`.
* **Handling:** If Carrier returns "All-In" rate, treat Surcharges as 0.

**FR-QUOTE-008: Volumetric Weight (Chargeable Weight)**
For LCL (Less than Container Load) requests, the system MUST calculate the Chargeable Weight.

* **Constants:** Sea Factor = 1000 kg/cbm.
* **Formula:** `VolumetricWeight = VolumeCBM * 1000`.
* **Logic:** `ChargeableWeight = MAX(ActualWeightKg, VolumetricWeight)`.
* **Usage:** The Carrier rate (Price per Ton/CBM) is applied to the ChargeableWeight.

**FR-QUOTE-009: Overweight Surcharge (OWS)**
The system MUST apply an internal surcharge if the carrier did not, but the weight exceeds thresholds.

* **Threshold:** 20FT > 21,000 kg.
* **Fee:** If Carrier response does not include code OWS, add internal cost `$150`.

### 3.4 Profit Margin Application

**FR-QUOTE-010: Margin Rules**
The system MUST apply profit margins on top of the `TotalCost`.

* **Step 1 (Percentage):** `SellPrice = TotalCost * 1.15` (15% Markup).
* **Step 2 (Fixed Fee):** `SellPrice = SellPrice + 25.00` (Handling Fee).
* *Note:* Margins must be applied after currency conversion.

**FR-QUOTE-011: Smart Tiering (Future Proofing)**
The system MUST support separate markup configurations based on the incoterm.

* If `incoterm == 'EXW'` (Ex Works - Complex), add extra `+ $50` handling fee.
* If `incoterm == 'CIF'` (Standard), add `+ $0`.

### 3.5 Output Generation

**FR-QUOTE-012: Response Sorting**
The system MUST sort the final list of quotes before returning.

* **Primary Sort:** Total Price (Ascending).
* **Secondary Sort:** Transit Time (Ascending).

**FR-QUOTE-013: ID Generation**
The system MUST generate a deterministic UUID for the quote based on the hash of the inputs + price. This allows the Booking Module to re-verify the price later.

## 4. Data Requirements

### 4.1 Caching Strategy (Redis)

**FR-DATA-001: Request Hashing**
The system MUST generate a SHA-256 hash of the request parameters (`origin` + `dest` + `weight` + `equipment`) to use as the Cache Key.

**FR-DATA-002: Cache Lifecycle**

* **Write:** Store successful `QuoteResponse` (Protobuf serialized or JSON) in Redis.
* **TTL:** Set expiration to **10 minutes**.
* **Read:** On new request, check Redis. If Hit -> Return immediately (skips calculation).

### 4.2 Redis Schema

```
REF:PORTS:{UNLOCODE}         -> Hash Map { "name": "Shanghai", "country": "CN", "zone": "Asia" }
RATES:EXCHANGE:{FROM}:{TO}   -> String "0.92"
CACHE:QUOTES:{SHA256_HASH}   -> String (JSON Blob)

```

## 5. Non-Functional Requirements (NFRs)

### 5.1 Performance

* **NFR-PERF-001:** Internal processing time (CPU calculation) MUST be **< 10ms** for a single request.
* **NFR-PERF-002:** Memory footprint per concurrent request MUST NOT exceed **2MB**.
* **NFR-PERF-003:** The service MUST support graceful shutdown, waiting for active calculations to finish (max 5s) before killing the pod.

### 5.2 Error Handling

* **NFR-ERR-001:** If the Carrier Mock returns a `5xx` error, the system MUST log the error and exclude that carrier from the results list. It MUST NOT fail the entire request unless *all* carriers fail.
* **NFR-ERR-002:** If the Carrier Mock returns a `429` (Rate Limit), the system MUST trigger the Circuit Breaker immediately.

### 5.3 Logging & Tracing

* **NFR-LOG-001:** All logs MUST be in structured JSON format.
* **NFR-LOG-002:** All logs MUST include the `transaction_id` passed in the gRPC metadata.
* **NFR-LOG-003:** The system MUST log the "Price Delta" (Raw Cost vs Sell Price) for analytics/audit purposes.

---

## 6. Edge Cases & Constraints

* **Zero Results:** If no carriers return a valid rate, return an empty list with status `OK`. Do not return an error. The UI will handle "No routes found".
* **Negative Margins:** The system MUST implement a safeguard preventing the `SellPrice` from being lower than `TotalCost` (even if bad configuration sets a negative markup).
* **Invalid Dates:** If `ReadyToLoadDate` is in the past, return `INVALID_ARGUMENT`.
